# Pass per-plan build/review through planner submission

## Architecture Context

The planner agent submits a plan set via the `submit_plan_set` MCP tool, which is wired with `planSetSubmissionSchema` from `packages/engine/src/schemas.ts`. Inside that submission, `orchestration.plans[]` entries are validated by `orchestrationPlanSchema` (lines 388-393). Today that schema only accepts `id`, `name`, `dependsOn`, `branch`. Zod's default `.strip()` silently drops any `build` or `review` the planner produces, even though the planner's prompt (`packages/engine/src/prompts/planner.md`) explicitly tells the model to choose per-plan build stages and review config.

`writePlanSet` in `packages/engine/src/plan.ts` (line 691-700) maps the validated submission entries into orchestration.yaml, currently emitting only `id`, `name`, `depends_on`, `branch`, and optionally `agents`. `injectPipelineIntoOrchestrationYaml` (line 754-777) then runs after the composer and backfills missing per-plan `build` and `review` from `pipeline.defaultBuild` / `pipeline.defaultReview`. The net effect is that every plan ends up sharing the same defaults — visible in the monitor UI as identical breadcrumbs (`implement > review-cycle > doc-update > test-write`) for every plan in every run.

The read side is already correct: `parseOrchestrationConfig`, the runner in `packages/engine/src/eforge.ts`, the daemon in `packages/monitor/src/server.ts`, and the monitor UI's `thread-pipeline.tsx` all read per-plan `build`/`review` properly. Only the planner-submission write path is broken.

## Implementation

### Overview

Extend `orchestrationPlanSchema` with optional `build` and `review` fields (reusing the existing `pipelineBuildStageSpecSchema` and `pipelineReviewProfileConfigSchema` from the same file). Update `writePlanSet` to conditionally include `build` and `review` per plan when present on the submission entry. Update the stale comment in `injectPipelineIntoOrchestrationYaml` to reflect that those fields are now planner-overridable. Add a unit test that exercises the planner-override path alongside the existing backfill test.

The schema YAML the planner sees is auto-generated by `getPlanSetSubmissionSchemaYaml()` from the Zod schema, so the new optional fields surface to the model with no separate prompt change. Making them optional preserves the existing backfill safety net.

### Key Decisions

1. **Reuse `pipelineBuildStageSpecSchema` and `pipelineReviewProfileConfigSchema` from `schemas.ts`** rather than introducing parallel definitions. They already model the exact shape needed and are co-located in the same file (lines 593, 606).
2. **Keep both fields optional**, not required. The existing backfill in `injectPipelineIntoOrchestrationYaml` is a working safety net and stays valid for any submission that omits them. Tests using `defaultBuild: ['implement', 'review-cycle']` continue to exercise the backfill path unchanged.
3. **Conditional spread in `writePlanSet`** mirrors the existing `agents` pattern at line 698-699 (`...(planData?.frontmatter.agents ? { agents: planData.frontmatter.agents } : {})`). Apply the same shape: `...(p.build ? { build: p.build } : {})` and `...(p.review ? { review: p.review } : {})`.
4. **No change to the planner prompt or `agents/planner.ts`.** The prompt already documents per-plan build/review, and `planner.ts` wires the schema generically so it picks up the new optional fields automatically.

## Scope

### In Scope

- Extend `orchestrationPlanSchema` in `packages/engine/src/schemas.ts` with optional `build` (using `pipelineBuildStageSpecSchema`) and `review` (using `pipelineReviewProfileConfigSchema`) fields.
- Update `writePlanSet` in `packages/engine/src/plan.ts` to conditionally pass through `build` and `review` from each orchestration plan entry.
- Update the stale comment at `packages/engine/src/plan.ts:766-768` to reflect that `build`/`review` are now planner-overridable with composer defaults as fallback.
- Add a new unit test in `test/plan-parsing.test.ts` (placed alongside the existing test at line 265) that round-trips a submission with per-plan `build`/`review` set on one plan and omitted on another.

### Out of Scope

- `packages/engine/src/prompts/planner.md` — already correct, do not modify.
- `packages/engine/src/agents/planner.ts` — wires schema generically, no change needed.
- `packages/engine/src/eforge.ts`, `packages/engine/src/pipeline/runners.ts`, `packages/monitor/src/server.ts`, `packages/monitor-ui/**` — read sides are already correct.
- Existing tests using `defaultBuild: ['implement', 'review-cycle']` — they exercise the backfill path and remain valid.
- Pipeline composer logic — no change to how defaults are produced.

## Files

### Modify

- `packages/engine/src/schemas.ts` — extend `orchestrationPlanSchema` (around line 388-393) with two new optional fields:
  - `build: z.array(pipelineBuildStageSpecSchema).optional().describe('Per-plan build stage pipeline; if omitted, the composer\'s defaultBuild is used as a backfill')`
  - `review: pipelineReviewProfileConfigSchema.optional().describe('Per-plan review configuration; if omitted, the composer\'s defaultReview is used as a backfill')`
  - Note: `pipelineBuildStageSpecSchema` and `pipelineReviewProfileConfigSchema` are declared later in the file (lines 593 and 606). To avoid a TDZ reference, either move those two const declarations above `orchestrationPlanSchema`, or re-declare equivalent inline schemas at the top of the file (preferred: move the existing declarations up — they have no upward dependencies).

- `packages/engine/src/plan.ts` — in `writePlanSet` (line 691-700), update the `payload.orchestration.plans.map(...)` callback to spread `build` and `review` conditionally, mirroring the existing `agents` spread:
  ```ts
  return {
    id: p.id,
    name: p.name,
    depends_on: p.dependsOn,
    branch: p.branch,
    ...(p.build ? { build: p.build } : {}),
    ...(p.review ? { review: p.review } : {}),
    ...(planData?.frontmatter.agents ? { agents: planData.frontmatter.agents } : {}),
  };
  ```

- `packages/engine/src/plan.ts` — update the comment at lines 766-768 inside `injectPipelineIntoOrchestrationYaml`. Replace the current text ("The planner's submission schema omits these (composer decisions), so writePlanSet emits orchestration.yaml without them...") with text that reflects the new contract: planner submissions may now include per-plan `build`/`review`, and this backfill fills in the composer defaults for any plan that omitted them so `parseOrchestrationConfig` always sees both fields populated.

- `test/plan-parsing.test.ts` — add a new test in the same `describe` block as the existing "backfills per-plan build/review from pipeline defaults when absent" test (around line 265). The new test must:
  1. Construct an `orchestration.yaml` representative of what `writePlanSet` would emit when the planner specifies per-plan build/review on plan A and omits them on plan B (i.e. plan A has `build` and `review` fields, plan B does not).
  2. Run `injectPipelineIntoOrchestrationYaml` against it with `ERRAND_PIPELINE`.
  3. Read back via `parseOrchestrationConfig`.
  4. Assert plan A's `build` and `review` exactly match the planner-chosen values (NOT `ERRAND_PIPELINE.defaultBuild`/`defaultReview`).
  5. Assert plan B's `build` equals `ERRAND_PIPELINE.defaultBuild` and plan B's `review` equals `ERRAND_PIPELINE.defaultReview` (backfill path still works).
  - Use distinct values for plan A's build/review (e.g. `[['implement', 'doc-update'], 'review-cycle']` and `{ strategy: 'parallel', perspectives: ['code', 'security'], maxRounds: 2, evaluatorStrictness: 'strict' }`) so they are guaranteed not to collide with `ERRAND_PIPELINE` defaults.
